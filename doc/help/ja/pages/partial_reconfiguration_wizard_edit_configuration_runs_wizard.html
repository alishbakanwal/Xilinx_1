[Edit Configuration Runs] ページ

ウィザードのこのページでは、デザイン内のさまざまな PR コンフィギュレーションのインプリメンテーション run を定義できます。さまざまなインプリメンテーションを使用して、コンフィギュレーションごとに複数の run を定義することで、どの結果が最適なのか確認できます。コンフィギュレーションのリコンフィギャラブル モジュール (RM) をインプリメントするためには、各 PR コンフィギュレーションに少なくとも 1 つの run が必要です。

このページには、プロジェクトのインプリメンテーション run を管理する [Configuration Runs] の表が表示されます。この表には、run が名前順にリストされます。 ユーザーは、コンフィギュレーションに run を割り当て、親 run を定義し、その run の制約ファイルとストラテジを指定できます。表が空の場合は、[automatically create configuration runs] というコマンド リンクが表示されます。このリンクをクリックすると、PR コンフィギュレーションに必要なインプリメンテーション run がすべて自動的に設定されます (推奨)。

デザインの各 RM がアウト オブ コンテキスト (OOC) 合成 run として別々に合成されてから、最上位デザインに統合されます。partitionDef または RM の作成後に [Design Runs] ウィンドウに表示されるように、合成は PR フロー用に自動的に設定されます。詳細は、『Vivado® Design Suite ユーザー ガイド : パーシャル リコンフィギュレーション』 (UG909) のこのセクションを参照してください。

インプリメンテーション run では各 RM の親を別々に設定できるので、コンフィギュレーションが完全なデザインとしてインプリメントされるか、RM だけが前にインプリメントされたデザインの内容にインプリメントされます。選択肢は次のとおりです。

  • [Synthesis (synth_1)] : 合成 DCP が RM に使用され、スタティック ロジックと RM が 1 つのデザインとして一緒にインプリメントされます。
  • [Implementation (impl_1)] : 親のインプリメンテーション結果がコンフィギュレーションに統合された RM および既存インプリメンテーション結果にインプリメントされた RM のネットリストと一緒に使用され、コンフィギュレーションが完了されます。

[Configuration Runs] の表のツールバー メニューには、次のコマンドが含まれます。

  • [Add] () : PR コンフィギュレーションに対して新しいインプリメンテーション run を定義できます。クリックすると、親 run、コンフィギュレーション名、スペースをリストする小さなダイアログ ボックスが開き、run の名前定義できます。デフォルトの run 名は指定されません。
  • [Remove] () : 現在のプロジェクトから選択したコンフィギュレーション run が削除されます。
  • [Show PartitionDef Instances] (): コンフィギュレーション内の partitionDefs および RM 割り当てをリストする列を切り替えることができます。

[Next] または [Skip to Finish] をクリックして Partial Reconfiguration ウィザードの [Partial Reconfiguration Summary] ページに進みます。
